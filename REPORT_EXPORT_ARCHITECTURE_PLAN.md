# JewelFlow — Report & Export Architecture Plan

> **Status: DESIGN ONLY.** No code, migrations, routes, templates, or exports were changed to produce this document. This is the architecture, UX, and output-design specification for a reworked reporting-export system. Implementation is a separate, later effort (see Rollout).
>
> **Date:** 2026-06-04
> **Scope:** How JewelFlow produces report outputs (screen, PDF, Excel, CSV) for shop owners, managers, accountants, CAs, GST consultants, auditors, banks, and government departments.

---

## 0. Decisions locked with the product owner

These were confirmed before this plan was written and constrain everything below:

| Decision | Choice |
|---|---|
| **PDF engine** | Headless Chromium renders dedicated print-only document templates to PDF (reuses the Chromium already installed on the server). NOT screen-HTML, NOT a PHP PDF lib. |
| **Rollout** | Build the shared spine, then prove it on **one report end-to-end first** — the **Sales / Invoice Register** — before templating across the rest. |
| **Branches** | Single-location today (no branch model exists). Architecture must **leave a branch hook** but ship no branch UI. |
| **Sensitive columns** | **CA-safe by default + permission-gated.** Sensitive columns are off by default in external/CA profiles; including them is an explicit per-export opt-in, allowed only for owner/manager. |

Guiding principle (owner's words): **the middle ground.** Not "print the screen," not a Salesforce/Power-BI report builder. A jewellery shop's reality stays the priority.

## 1. Current-state assessment

### 1.1 What exists today

**Two generations of reports coexist:**

- **Legacy screen+print** — `gst`, `closing` (daily closing), `daily`, `cash`, `pnl`, `gold`, `repairs`, `metal-exchange`. These render a Blade screen with an `@media print` block. "PDF" = the user pressing Ctrl+P / Save-as-PDF on the screen. Output quality depends on the browser and reads like a screen capture, not a document.
- **Newer `Reporting/*` controllers** — `TaxReportController` (GSTR-1, GSTR-3B, CN register), `ReconciliationReportController` (payment reconciliation, day book, inventory valuation, dead stock, purchase efficiency), `KarigarReportController` (settlement, shrinkage), `AuditReportController` (operator performance, suspicious activity), `ReceivablesReportController` (dues aging, EMI, scheme liability, metal liability). Most of these already have a `…/export` route that streams **raw CSV via `fputcsv`**.
- **A bulk Export Center** — `ExportController` (`/export`): customers, products, invoices, gold-ledger, cash-transactions, "export all" (manufacturer). Also raw `fputcsv` streams, gated by `reports.export`.

**Libraries / infra:**

- `maatwebsite/excel` (PhpSpreadsheet) **is installed in composer** but **not used by the report exports** — they all use raw `fputcsv`.
- **No PDF library** (no dompdf / mpdf / wkhtmltopdf / snappy). PDFs are browser-print only.
- Headless **Chromium is available** on the server (Playwright build) — currently used only for visual QA, not for document generation.
- Document-header data exists on `shops` / `shop_billing_settings`: legal name, `address_line1/2`, `city`, `state`, `state_code`, **GSTIN**, bank details.

**Permissions today:** `reports.view` (most reports), `reports.daily_closing` (closing), `reports.export` (Export Center), plus `edition:retailer|manufacturer` middleware. Everything is tenant-scoped to the shop via the global scope.

### 1.2 Concrete weaknesses

1. **PDF = screen capture.** Print output is the screen DOM with a print stylesheet — no formal header/footer, no GSTIN block, no "generated by/at", no page numbering, inconsistent table pagination. Not acceptable to hand to a CA, bank, or government office.
2. **CSV bakes presentation into data.** Current CSVs interleave banner rows (`== B2B ==`), blank rows, and label rows with the data. That is neither machine-friendly (breaks `pd.read_csv`, breaks GSTN offline-tool import) nor analysis-ready.
3. **No Excel proper.** Despite `maatwebsite/excel` being installed, reports never produce a typed `.xlsx` (numbers-as-numbers, dates-as-dates, frozen headers, auto-filter). Accountants who want to pivot get a CSV with rupee symbols and thousands separators as text.
4. **No export-scope control.** A report shows "Today"; to export "Last financial year" the user must change the on-screen filter, re-run, then export — and the export never records which filters produced it.
5. **No column control.** All-or-nothing. An owner cannot strip customer mobile / operator / cost before sending a sales report to their CA.
6. **Inconsistent export surface.** Some reports stream CSV; legacy ones only print. No shared document furniture, no shared filter UX, no shared permission model for "who can export what."
7. **No saved presets.** "The monthly CA export" must be reconstructed by hand each month.

### 1.3 What is already good (keep)

- Tenant scoping is automatic and safe.
- The newer `Reporting/*` controllers cleanly separate the **dataset query** (in service/controller methods like `tax->gstr1()`) from the screen — a good base to build the shared dataset layer on.
- Compliance data is already correct at source (CGST/SGST split, place of supply, HSN, credit-note reversal). The problem is **presentation**, not calculation.

## 2. Best-practice research (Indian SMB + ERP context)

How comparable tools handle exports, and what is appropriate to borrow for a jewellery shop:

| Tool | Export model | What's worth borrowing | What to avoid |
|---|---|---|---|
| **Tally Prime** | Fixed report layouts; export to PDF/Excel/XML/JSON; "Configure (F12)" toggles a **fixed set of columns/options** per report; period is a first-class, always-visible control. | Fixed column toggles (not a builder); period always front-and-centre; formal print layout with company header. | Dense, dated UI; steep learning curve. |
| **Vyapar** (India SMB) | One-tap PDF/Excel; PDFs are **proper business documents** with firm header, GSTIN, totals; share-to-WhatsApp. | Owner-friendly one-tap formal PDF; document furniture by default; share flow. | Limited column control; limited compliance depth. |
| **Busy / Marg** (Indian accounting) | Strong statutory/GST formats; rigid compliance outputs matched to government formats; Excel for working data. | Treat compliance outputs as **rigid, format-locked**; separate "working Excel" from "filing format". | Old-school UX. |
| **Zoho Books / QuickBooks** | Report = data + a small set of presentation options; **saved customizations/presets** (named, shareable); scheduled email exports; clean Excel + CSV; PDF with org header. | Named **presets** as the unit users save and reuse; clean Excel/CSV separation; org-branded PDF. | Their full custom-report builder is overkill here. |
| **Salesforce / Power BI** | Drag-and-drop report/dashboard builder, custom SQL/DAX. | (Conceptual only) the value of saved, shareable definitions. | **Do not replicate.** A jewellery shop will never build reports this way. |

**Synthesis for JewelFlow:** the winning SMB pattern is **fixed reports + a small, system-defined set of presentation choices (profile + optional columns + format) + named presets + a genuinely formal PDF**. That is exactly the "middle ground" — closest to Vyapar's owner-friendliness plus Zoho's presets plus Tally/Busy's rigid statutory formats, minus any builder.

## 3. Export architecture (answers Q1)

### 3.1 Shape: shared dataset, independent renderers

The single most important decision. Screen / PDF / Excel / CSV are **NOT independent** (that causes number drift and duplicated query logic) and are **NOT one shared HTML** (that causes screen-capture PDFs). Instead:

```
┌──────────────────────────────────────────────────────────────────┐
│ ReportDefinition (per report, declarative metadata)               │
│   • columns: key, label, type, tier (mandatory|optional|sensitive)│
│   • profiles: Summary | Detailed | CA | Raw (column sets + opts)  │
│   • filters: which filter controls apply                          │
│   • classification: owner|operational|accounting|compliance|audit │
│   • permissions: view + export + sensitive gates                  │
│   • formats: which of screen/pdf/excel/csv are allowed            │
└───────────────────────────────┬──────────────────────────────────┘
                                 │ resolved filters + profile + column selection
                                 ▼
              ┌──────────────────────────────────────────┐
              │ Report Dataset Service (one per report)   │
              │  → canonical RowSet:                      │
              │     • sections[] (for multi-part reports) │
              │     • columns[] (resolved, ordered)       │
              │     • rows[] (typed values)               │
              │     • totals / subtotals                  │
              │     • meta (filters applied, period,      │
              │            shop, generated_by/at)         │
              └───────────────┬──────────────────────────┘
                              │  (single source of truth — no renderer re-queries)
        ┌─────────────┬───────┴────────┬──────────────────┐
        ▼             ▼                ▼                  ▼
   Screen view    PDF renderer     Excel writer       CSV writer
   (Blade,        (Chromium +      (maatwebsite,      (clean single
   paginated,     report-document  typed cells,       table, raw
   interactive)   print template)  sheets per         values)
                                   section)
```

**Rules:**
- A renderer **must consume the dataset service output**; it must never run its own query. This guarantees screen and exports always agree.
- The **dataset is presentation-agnostic** (typed values, not formatted strings). Formatting (₹, thousands separators, date format) is each renderer's job — so CSV/Excel get raw numbers and PDF/screen get pretty ones.
- `ReportDefinition` is **declarative metadata**, so adding/maintaining reports is filling in a definition, not writing four exporters.

### 3.2 The four layers (validates the owner's model, refined)

- **Layer 1 — Filters** (*what data*): date range, operator, customer, metal type, status. **No branch** (hook reserved). See §6.
- **Layer 2 — Export Profile** (*how it's shaped*): a named, system-defined preset of {column set + grouping/subtotals + sensible default format}. Standard set: **Summary / Detailed / CA / Raw** (a report may expose a subset). The profile is the primary "how" control. See §8.
- **Layer 3 — Format**: PDF / Excel / CSV (and Screen). A profile declares which formats make sense; the user picks one.
- **Layer 4 — Optional columns**: fine-grained toggles **within the profile's allowed catalog**. Not free-form. See §7.

Refinement vs the original 4-layer sketch: **Profile drives the defaults** (column set + default format); columns are the fine adjustment inside a profile; this prevents the UI from feeling like a builder while still giving control.

## 4. PDF architecture (answers Q5)

### 4.1 Engine

A shared service renders an HTML **print template** (server-side Blade, *not* the screen view) to PDF using the installed **headless Chromium** (`--no-sandbox` since the app runs as root; `--disable-dev-shm-usage`). Chromium gives exact CSS fidelity (proper tables, page-break control, ₹ glyph, web fonts) that PHP PDF libs struggle with. Generation runs **synchronously for normal sizes**, with a **queued path for large documents** (see Risks).

### 4.2 The `report-document` layout (reusable furniture)

Every report PDF uses one shared document layout; the report only supplies its title + sections + columns.

**Mandatory header (every page or first page + running header):**
- Shop **legal name**
- Shop address (line1/line2, city, state)
- **GSTIN** (and shop state code)
- Report **name**
- Report **period** (e.g., "1 Apr 2025 – 31 Mar 2026")
- **Filters applied** (echoed, human-readable: "Operator: all · Status: finalized")
- **Profile** used (e.g., "CA Format")

**Mandatory footer (every page):**
- **Generated by** (user name)
- **Generated at** (timestamp + timezone)
- **Page X of Y**
- "System-generated report — JewelFlow" line
- Optional: a short document reference/serial for traceability

**Body standards:**
- Title block, then section tables.
- **Table headers repeat on every page**; rows never split across a page break.
- Subtotals per section + a clear grand total row.
- Numeric columns right-aligned, `tabular-nums`, ₹ + Indian grouping.
- Consistent typography (reuse the teal/hairline system already standardised across the app for on-brand, professional output).

**Mandatory vs optional furniture:** everything in header/footer above is **mandatory** except logo (optional) and document serial (optional). The filters-applied echo is mandatory — it is what makes the document trustworthy ("this is exactly what was queried").

## 5. Excel & CSV architecture (answers Q6)

The three formats have **distinct jobs** and distinct structures:

### 5.1 PDF — human-readable (covered in §4)
Formal document for people and institutions.

### 5.2 Excel — analysis-ready (`maatwebsite/excel`)
- **Data sheet = one clean table.** Header row, then data rows. **No banner rows, no blank rows in the data region.**
- **Typed cells:** numbers as numbers (not "₹1,23,456.00" text), dates as real dates, percentages as numbers. This is what lets an accountant pivot/sum.
- **Frozen header row + auto-filter** on the table.
- A clearly marked **total row** (or a separate totals area below a blank gap, never interleaved).
- **Multi-section reports** (e.g., GSTR-1 B2B / B2CS / HSN) → **one sheet per section**, each a clean table.
- A **"Report Info" sheet** (or a small top band on sheet 1, clearly separated from the table) carrying the same metadata as the PDF header: shop, GSTIN, report, period, filters, generated by/at, profile.

### 5.3 CSV — machine-friendly
- **One table per file.** Single header row, then rows.
- **Raw values:** no ₹, no thousands separators, ISO `YYYY-MM-DD` dates, decimals with `.`, booleans as `true/false` or `1/0` consistently.
- **No banners, no blank rows, no label rows.** (This is the explicit fix for today's CSVs.)
- Multi-section reports → either **separate CSV files in a ZIP**, or a leading **`section` column**. Default: ZIP of per-section CSVs for compliance reports (matches GSTN offline-tool expectations); single-table reports stay single CSV.
- Metadata does **not** go inside the CSV (machines don't want it). If needed, ship a sidecar `…-meta.txt`/`.json` in the ZIP.

**Rule of thumb:** PDF for sending to a human/institution; Excel for the accountant who will work the numbers; CSV for importing into another system (GSTN tool, Tally, a bank's template).

## 6. Filter architecture (answers Q2)

### 6.1 Page filters vs export filters — one model, no hidden layer

**Do not** create a separate, invisible export filter layer (that is the confusion trap). Instead:

1. The report screen has its filters (date range, operator, customer, metal, status).
2. **Export is an explicit step** (an "Export" panel/drawer/modal) that **pre-fills from the current page filters**.
3. In that panel the user can **change the scope before generating** — including quick presets ("This month", "Last month", "Last financial year", "Custom"). So viewing "Today" but exporting "Last FY" is a deliberate, visible choice in the panel.
4. The **generated document stamps the exact filters used** in its header (PDF) / info sheet (Excel) / sidecar (CSV).

Net effect: what you see, what you export, and what the document says it contains are always reconciled — no silent mismatch.

### 6.2 Filter catalogue

Common filters (a report opts into the subset it supports via `ReportDefinition`):
- **Date range** (mandatory for transactional reports) with FY-aware quick presets.
- **Operator** (who handled the transaction).
- **Customer**.
- **Metal type** (gold/silver/platinum…).
- **Status** (draft/finalized/void, etc.).
- **Branch** — **reserved hook, not rendered** (single-location today). When multi-branch arrives, it slots in as another filter + a dataset dimension with no architectural rework.

## 7. Column-customization model (answers Q3)

A **fixed, system-defined catalogue per report** — never a free-form builder.

### 7.1 Three tiers (declared in `ReportDefinition`)

| Tier | Behaviour |
|---|---|
| **Mandatory** | Always present; cannot be removed (document integrity — e.g., invoice no., date, taxable, total). |
| **Optional** | User toggles within the allowed set per export (e.g., HSN, payment mode, discount). |
| **Sensitive** | Customer mobile, operator name, internal notes, **cost/margin**. **Off by default in CA/external profiles**, opt-in per export, **permission-gated to owner/manager**. |

### 7.2 Behaviour
- The profile sets the **default** column selection; the user fine-tunes only within the catalogue.
- Sensitive columns require both: (a) an explicit toggle in the export panel, and (b) the `export sensitive` permission. A staff member with plain `reports.export` simply cannot include them — the toggles are hidden/disabled.
- Compliance reports expose **no** column toggles at all (§9).

This satisfies "owners can strip mobile/operator before sending to the CA" without becoming a report builder.

## 8. Preset architecture (answers Q4)

Two kinds:

1. **System profiles** — the built-in Summary / Detailed / CA / Raw (and compliance formats). Undeletable, ship with the app, define the column-set + format defaults.
2. **Saved presets** — a saved, named combination of **{filter scope + profile + column selection + format}**. Examples: "Monthly CA Export", "GST Filing Export", "Branch Audit Export".

**Scope recommendation: shop-wide, owner/manager-created.**
- A jewellery shop operates as one unit — "the CA export" is a shop thing, not a per-user thing. Shop-wide presets mean staff reuse the owner's blessed definitions; no per-user clutter.
- **Who can create/edit shop-wide presets:** owner/manager (`reports.export` + a manage flag). Everyone with export rights can **use** them.
- **Personal presets:** deferred (YAGNI for now). The model leaves room (a `scope: shop|user` column) if demand appears.
- A preset that includes sensitive columns still enforces the sensitive permission **at export time** — saving a preset never bypasses the gate.

## 9. Compliance constraints (answers Q7)

**Compliance reports are rigid. No exceptions.**

- **GST Summary, GSTR-1, GSTR-3B (support), Credit/Debit Note Register, and any statutory/tax report** = **fixed columns, fixed layout, no optional/sensitive/custom columns, no profile choice** beyond the prescribed format.
- Only the **period** (and where legally relevant, GSTIN/place-of-supply context) varies.
- Outputs: the **formal PDF** (for the CA/owner to read & file-reference) **plus** a **prescribed CSV/Excel** structure aligned to the GSTN offline tool / standard filing layout.
- Rationale: government and CAs expect the canonical shape; allowing column edits invites filing errors and undermines trust. Correctness and format-fidelity outrank flexibility here.

The flexible machinery (profiles, optional columns, presets) applies to **non-compliance** classes only.

## 10. Permissions model (answers Q8)

Layered on the existing permission system, tenant-scoped automatically:

| Capability | Gate |
|---|---|
| View a report on screen | `reports.view` (+ `reports.daily_closing` for closing) |
| Export any report (PDF/Excel/CSV) | `reports.export` |
| Include **sensitive** columns / **cost-margin** / **raw-data** exports | higher gate — **owner/manager** (e.g., `reports.export_sensitive`); off otherwise |
| Create/edit **shop-wide presets** | owner/manager |
| **Audit-class** reports (suspicious activity, operator performance) | owner/manager only — not visible to floor staff, not customer-shareable |
| **Compliance** exports | `reports.view` is sufficient (CAs/managers need them); optionally a `reports.compliance` if a shop wants to restrict |

Principle: **everyone does not export everything.** Plain staff can run and export operational reports they're allowed to see, but sensitive columns, raw dumps, and audit reports require elevation.

## 11. Report classification (answers Q9)

Every report maps to one of five classes; the class sets its export rules.

| Class | Reports (current) | Column flexibility | Formats | PDF style | Access |
|---|---|---|---|---|---|
| **Owner** | P&L, Gold balances, dashboards, margin views | Flexible; **cost/margin gated** | PDF + Excel + CSV | Management styling | owner/manager |
| **Operational** | Dead stock, Karigar settlement, Shrinkage, Purchase efficiency, Repairs, Metal exchange | Moderate (optional cols) | **Excel-leaning** + PDF | Standard | `reports.view`/`export` |
| **Accounting** | Day Book, Payment reconciliation, Cash flow, Daily closing, Inventory valuation | Limited | **Formal PDF mandatory** + Excel | Strong document furniture | `reports.view`/`export` |
| **Compliance** | GST Summary, GSTR-1, GSTR-3B, Credit Note Register | **None (rigid)** | Formal PDF + prescribed CSV/Excel | Statutory/formal | `reports.view` |
| **Audit** | Suspicious activity, Operator performance | Full detail | Excel + PDF | Plain | **owner/manager only** |
| **Receivables** (sub-set of Accounting/Owner) | Customer dues aging, Pending EMI, Scheme liability, Metal liability | Moderate; customer PII sensitive | PDF + Excel | Standard | `reports.view`/`export` |

**Per-class summary of the four levers:**
- *Export flexibility:* highest for Owner/Operational, lowest for Compliance.
- *Column flexibility:* none for Compliance, gated for Owner (cost/margin) and any class exposing customer PII, otherwise optional-tier.
- *Profile requirement:* Compliance = single fixed profile; others = Summary/Detailed/CA/Raw subset.
- *Format requirement:* Accounting & Compliance **must** offer the formal PDF; Operational/Audit lead with Excel; all transactional reports offer clean CSV.

## 12. Rollout recommendation

**Phase 0 — Build the spine (no user-visible reports change yet).**
- `ReportDefinition` metadata structure + registry.
- `ReportDataset` service contract (canonical RowSet with sections/columns/rows/totals/meta).
- The four renderers: Screen adapter, **Chromium PDF renderer + `report-document` layout**, Excel writer (maatwebsite, typed), CSV writer (clean).
- The **Export panel UX** (pre-filled, editable scope; format + profile + column toggles; sensitive gating).
- Column-tier + permission gates; preset storage (shop-wide).
- Chromium PDF service hardening (root/`--no-sandbox`, sync + queued path).

**Phase 1 — Pilot: Sales / Invoice Register, end-to-end.**
Chosen because it exercises the *most* machinery: full filter set, all four profiles, optional **and** sensitive columns (mobile/operator/cost), and all three formats with the formal PDF. Ship it, get real CA/owner feedback, refine the spine.

**Phase 2 — Compliance set on the rigid track.**
GSTR-1, GSTR-3B, Credit Note Register, Day Book — formal PDF + prescribed CSV/Excel, period-only filter, no column flex. Highest external-trust value.

**Phase 3 — Migrate the rest.**
Bring legacy screen+print reports (`gst`, `closing`, `daily`, `cash`, `pnl`, `gold`, `repairs`, `metal-exchange`) and remaining `Reporting/*` reports onto the spine. **Retire ad-hoc `fputcsv`** and the banner-CSV format. Keep the Export Center but route it through the same renderers.

**Branch:** ships as an unused hook throughout; activated only if/when a multi-location data model exists.

## 13. Risks & mitigations

| Risk | Mitigation |
|---|---|
| **Chromium PDF infra** (root sandbox, missing libs, ₹/font glyphs, concurrency) | Run with `--no-sandbox --disable-dev-shm-usage`; pin a known Chromium; embed/serve fonts that include ₹; **queue large documents** and notify on completion; cap concurrent renders. |
| **Memory on big Excel/CSV** (financial-year raw dumps) | Chunked/streamed writes; queue large exports; row-count guardrails with a "this is large, we'll email it" path. |
| **Breaking existing CSV consumers** | The banner-CSV cleanup changes output shape; inventory who consumes current CSVs (CAs, scripts) and provide the new clean format as the default with a brief migration note. |
| **Scope creep toward a builder** | Hard guard: **fixed column catalogue per report**, system-defined profiles only. No free-form column/SQL/expression input, ever. |
| **Renderer bypassing the dataset service → number drift** | Architectural rule + review: renderers consume `ReportDataset` output only; no renderer may query. Screen and exports share the dataset. |
| **Sensitive-column leakage** | Default-off in external profiles + permission gate enforced at *export time* (not just UI), so presets/links can't bypass it. |
| **Compliance format drift** | Compliance outputs are locked to the prescribed structure with tests against the expected layout; no column toggles exposed. |
| **Over-building before validation** | The pilot-first rollout exists precisely to validate the spine on one report before spreading cost across ~24 reports. |

---

## 14. Final recommendation

Adopt a **shared-dataset / independent-renderer** architecture driven by declarative **`ReportDefinition`** metadata, with:

- **One dataset service per report** as the single source of truth (screen and exports can never disagree).
- **Four renderers** — interactive screen, **Chromium-rendered formal PDF**, typed analysis-ready **Excel**, clean machine **CSV** — each with the right job, never screen-captures.
- A **four-layer control model** — filters (what) → profile (how) → format → optional columns (fine) — that feels like Vyapar/Zoho, not Salesforce.
- An **explicit, pre-filled, scope-editable export step** that stamps the exact filters onto every document.
- A **three-tier column model** (mandatory / optional / sensitive) from a fixed catalogue, with **sensitive columns CA-safe-by-default and permission-gated**.
- **Shop-wide named presets** (owner/manager-created) for repeatable exports like "Monthly CA Export".
- **Rigid, format-locked compliance reports**; flexibility only where it's safe.
- A **permission ladder** so not everyone exports everything.
- **Pilot on the Sales/Invoice Register**, then compliance, then migrate the rest; a **branch hook** left dormant.

This delivers professional, CA-friendly, government-friendly, owner-friendly output that is flexible where it should be and rigid where it must be — without turning JewelFlow into a BI platform or a report builder.

# ADDENDUM A — Refinements & Gap Analysis (2026-06-04)

> Approved in principle; this addendum adds versioning, export audit, FY-first UX, the locked CA Standard profile, watermarking, large-export strategy, preset evolution, and the **full report matrix**. Still **design only** — no code, migrations, templates, exports, or commits.

## 15. Report versioning (item 1)

Every generated export carries a **provenance block** so a CA or owner comparing two copies of "the same" report from different dates can tell *why* they differ.

**Five stamped fields (mandatory on every export, all formats):**

| Field | Meaning | Source |
|---|---|---|
| **Report Version** | Version of the report's *logic + layout contract* | `ReportDefinition.version` |
| **Generated At** | Timestamp + timezone of generation | server clock |
| **Generated By** | User who ran the export | auth user |
| **Export Profile** | Which profile produced this shape (e.g., "CA Standard v2") | selected profile |
| **Filters Applied** | The exact resolved filter set | export request |

**What "Report Version" means (precise):** a monotonic version string per report (e.g., `gstr1@3`) that bumps when **any of these change**: the dataset/query logic, the column contract, the totals/grouping rules, or the document layout. It does **not** bump for data changes (new invoices) — only for *how the report is computed or presented*. Profiles carry their **own** version (`CA Standard v2`) so a profile tweak is distinguishable from a logic change.

**Why split report-version from profile-version:** the four comparison questions the owner raised map 1:1 —
- "did report logic change?" → Report Version differs
- "did export profile change?" → Profile version differs
- "did filters change?" → Filters Applied differs
- "did document generation change?" → a `generator` build tag (renderer version) in the footer

**Placement:** PDF footer/last-page provenance panel; Excel "Report Info" sheet; CSV sidecar `…-meta.json`. The block is identical across formats so two exports are diffable field-by-field.

**Storage:** versions live in `ReportDefinition` metadata (code-defined, not a DB table) so they're reviewable in version control. A changelog comment per report records what each bump changed. **Design only — no migration.**

## 16. Export audit trail (item 2)

A **metadata-only** record of every export. **No exported file is ever stored** (privacy + storage; the file is reproducible from the stamped fields).

**Captured per export event:** user, timestamp, report (key + version), profile (+ version), format, resolved filters (JSON), row-count, sync-or-queued, success/failure, and whether **sensitive columns were included** (important — see below).

| Concern | Recommendation |
|---|---|
| **Storage** | A dedicated append-only `report_exports` audit table (metadata rows only). Reuse the existing AuditLog pattern/immutability discipline; never store the document blob. |
| **Retention** | Keep **24 months** by default (covers two FY cycles + a filing season). Older rows archived/pruned by a scheduled job. Configurable per shop later. |
| **Permissions** | Owner/Manager can view the shop's export audit. Plain staff cannot. Cross-shop access never (tenant-scoped). |
| **Visibility** | A read-only "Export history" view (filter by user/report/date). Surfaces *who exported what, when, with which filters, and whether sensitive data was included*. |
| **Audit usage** | (a) Data-governance — see who pulled customer PII / cost-margin / raw data; (b) reproduce a CA's copy exactly (same report-version + profile + filters); (c) detect anomalies (e.g., a staff member mass-exporting customer lists). |

This pairs with the sensitive-column gate (§7/§10): the gate prevents unauthorized inclusion; the audit records every authorized inclusion.

## 17. Financial-year-first filter UX (item 3)

Indian jewellery accounting is FY-centric (Apr 1 – Mar 31). The date control leads with **first-class presets**, not a raw calendar.

**Standard preset set (all transactional reports):**
`Today` · `This Month` · `Last Month` · `This FY` · `Last FY` · `Custom`

**Additional FY-oriented presets to include (audit result — yes, add these):**
- **`This Quarter` / `Last Quarter`** — GST/advance-tax cadence is quarterly; common for review.
- **`FY to date`** — Apr 1 → today (the most-used "where are we this year" view).
- **A specific-FY picker** in Custom (e.g., "FY 2024–25") — so a CA can pull a closed year by name without computing dates. Presets resolve to explicit dates which are then **stamped** in the document (§15), so "This FY" in the panel always prints as "1 Apr 2025 – 31 Mar 2026".

**Rules:** the panel **defaults to the report's natural period** (compliance → current GST month; closing/day-book → today; owner/operational → This Month). The FY presets are computed from the shop's FY start (fixed Apr 1 for India; left as a setting hook in case of edge cases). Month/quarter boundaries are timezone-correct.

## 18. CA Standard — locked profile (item 4)

**Recommendation: YES — introduce `CA Standard` as a distinct, system-locked profile, separate from the per-report "CA" profile.**

| Aspect | "CA" profile (existing) | **"CA Standard" (new, locked)** |
|---|---|---|
| Who defines it | Per report, may expose a few optional toggles | **System-maintained, identical across all shops** |
| Editable | Columns can be toggled within the catalogue | **Locked — no column/format edits, no per-shop variation** |
| Purpose | "CA-leaning" convenience shape | A **canonical, reproducible** shape a CA can rely on across every JewelFlow shop |
| Sensitive columns | Off by default, opt-in possible | **Never includes sensitive columns** (cost/margin/internal notes); customer PII limited to what statute needs |
| Versioning | profile version | **profile version, centrally bumped**; the version is printed so CAs know the format generation |

**Why it should exist:** CAs handle many shops. A *locked, identical-everywhere* CA Standard means a CA learns one layout and trusts it; reconciliation across clients is consistent; and the shop owner cannot accidentally produce a non-standard "CA" file. The flexible per-report "CA" profile stays for owners who want a CA-ish but tweakable export. So: **two tiers — `CA Standard` (locked, canonical) and `CA` (flexible).**

**Scope:** `CA Standard` applies to Accounting + Compliance + the CA-relevant Owner reports (Sales register, Day Book, GST set, Dues, P&L summary). Operational/Audit reports don't get a CA Standard (not CA-facing).

## 19. Internal / Draft watermarking (item 5)

**Recommendation: YES — document labels/watermarks driven by classification + state, not free-form.**

**Three system labels:**
- **`DRAFT`** — applied when the underlying data is not yet finalized for the period (e.g., a period that isn't closed, or a report run mid-day) **or** when the user explicitly marks a preview. Diagonal light watermark + header tag.
- **`INTERNAL USE ONLY`** — applied automatically to reports/columns not meant to leave the business.
- **`CONFIDENTIAL`** — applied to high-sensitivity financials.

**Classification rules (automatic, not user-chosen):**

| Report / condition | Label |
|---|---|
| **Margin / cost reports**, **P&L with cost**, **Purchase efficiency** | `CONFIDENTIAL` |
| **Audit reports** (suspicious activity, operator performance) | `INTERNAL USE ONLY` + `CONFIDENTIAL` |
| **Operator performance**, internal operational reports | `INTERNAL USE ONLY` |
| Any export where **sensitive columns were opted in** | `CONFIDENTIAL` (regardless of base report) |
| **Compliance / CA Standard** exports | **No watermark** (these are meant to be shared formally; a watermark would look unprofessional/invalid to a government office) |
| Period **not finalized / closed** | `DRAFT` overlay |

**Behaviour:** watermark renders in PDF (diagonal, low-opacity, non-obscuring) and as a bold banner line in Excel's Info sheet / CSV sidecar. The label is **not removable by floor staff**; only the classification/state controls it. This makes "don't share this outward" visible on the document itself.

## 20. Large-export strategy (item 6)

Expanded from the original risk note into an explicit model.

**Decision flow by estimated row-count (estimated before generation from the filtered query count):**

| Size band | Path | UX |
|---|---|---|
| **Small** (≲ 2,000 rows / single period) | **Synchronous** | Inline spinner; file downloads in the response. Covers the overwhelming majority of shop exports. |
| **Medium** (~2,000–20,000) | **Synchronous with progress**, or queued if PDF (Chromium render is the cost) | Spinner + "preparing…"; CSV/Excel stay sync, large **PDF** goes queued. |
| **Large** (> 20,000 rows, multi-year, full raw dumps) | **Queued job** | "We're preparing your export — you'll be notified when it's ready." |

**Thresholds:** row-count is the primary trigger; **PDF has a lower threshold** than CSV/Excel because Chromium rendering of thousands of paginated rows is the expensive operation. Exact numbers are tunable settings, not hard-coded.

**Queued export lifecycle:**
1. User requests → job enqueued → audit row written (status `queued`).
2. Worker generates the file to **temporary storage** with a **signed, expiring download URL**.
3. User notified (in-app notification + optional email) with the link.
4. **Download lifecycle:** file auto-expires (default **7 days**), then is deleted by a scheduled sweep. The **audit row persists** (metadata only) even after the file is gone — consistent with §16 ("never store exported files" long-term; the queued file is a transient artifact, not an archive).
5. Re-download within the window is allowed; after expiry the user re-runs (reproducible via stamped fields).

**Specific cases:**
- **Financial-year exports** — usually Medium; sync for CSV/Excel, queued for the formal PDF.
- **Multi-year / "export all"** — always queued; offered as Excel/CSV (a multi-year *formal PDF* is discouraged — too large to be a useful document; offer per-FY PDFs instead).
- **Very large invoice datasets** — queued; chunked/streamed writes (maatwebsite chunk reading) to bound memory.

**Concurrency:** cap simultaneous Chromium renders; queue beyond the cap (protects the box).

## 21. Preset evolution (item 7)

**Recommendation: ship shop-wide presets now; make the schema user-scope-ready; do NOT build user presets yet.**

- **Now:** presets are **shop-wide**, owner/manager-created, everyone-with-export-rights uses them. Matches how a shop actually works ("the" monthly CA export).
- **Architecture-ready:** the preset record carries a **`scope` field (`shop` | `user`)** and an **`owner_id`** from day one, even though only `shop` is written/exposed initially. Adding user presets later = enable the `user` scope + a "My presets" UI; **no migration, no rework**.
- **Justification gate (why not now):** a single-location shop with a few staff has no demonstrated need for private per-user presets; adding the UI now is clutter (YAGNI). The cost of being ready-but-not-shipping is one nullable column + an enum value. Revisit only if multi-staff shops ask for private presets.
- **Guardrail:** a shop-wide preset that includes sensitive columns still enforces the sensitive permission **at export time** for the running user — a preset never elevates privileges.

# ADDENDUM B — Dhiran (Gold-Loan / Pawn) Report Family Audit (2026-06-04)

> Focused audit requested before Phase-0 planning. **Design only — no code, templates, exports, or commits.** Grounded in the actual `App\Models\Dhiran\*` models and `DhiranController` report methods.

## 23. Verdict: same spine, dedicated *family* (not a separate system)

**Dhiran reports use the SAME export architecture** (one dataset service → independent PDF/Excel/CSV/screen renderers, `ReportDefinition` metadata, FY-first filters, versioning §15, export audit §16, large-export strategy §20). **Do not build a parallel export system.**

**But they must be registered as a distinct report family** with family-level defaults, because they differ from sales/inventory reports on three axes the rest of the app doesn't hit:

1. **Different domain entity** — loans, pledges, payments (not invoices/stock). The dataset contract must serve loan data (already flagged as Phase-0 requirement #1).
2. **Highest-sensitivity PII in the product** — `kyc_aadhaar`, `kyc_pan`, `kyc_photo_path`, plus pledged-valuables detail and customer identity. This is stricter than the customer-mobile sensitivity elsewhere.
3. **Regulatory/legal document needs** — pawnbroking/gold-loan activity in India is governed (state Pawnbrokers Acts; KYC/AML; Aadhaar handling rules; forfeiture/auction notice requirements; interest-rate transparency). Some outputs are **legal documents**, not tabular reports.

**Family-level defaults (apply to all Dhiran reports):**
- Permission baseline: `dhiran.reports` (exists). KYC fields and forfeiture detail require elevation above plain `dhiran.reports`.
- **Aadhaar/PAN masked by default** (`XXXX-XXXX-1234`); full value is a sensitive-tier, owner/manager-gated, **audited** reveal — and **never** emitted in a shareable CSV/Excel by default.
- Formal **PDF mandatory** (these face borrowers, auditors, sometimes courts/regulators).
- Default watermark posture: `CONFIDENTIAL` (pledged-asset + KYC data).
- Customer photo (`kyc_photo_path`) is never placed in list/Excel/CSV exports; only on a per-loan document where justified.

## 24. Two bespoke PDF layouts the family needs (beyond the standard report-document)

Most Dhiran reports are list-style and use the shared `report-document` layout (§4). **Two outputs are documents, not lists**, and need their own templates (same Chromium renderer, different Blade template — the architecture already allows a `ReportDefinition` to name a custom PDF template):

- **Loan Account Statement** (per loan) — principal, interest accrued, payment history, running balance, pledged-item detail, rate-at-pledge. A borrower/auditor-facing statement like an invoice, not a list. *(Borderline "report"; host it in the same renderer family.)*
- **Forfeiture Notice / Forfeiture Record** — the legally sensitive document: customer, pledged item + valuation, `forfeiture_notice_sent_at`, `forfeiture_notice_text`, dates. This is the audit/legal evidence of a lawful forfeiture and needs a formal, fixed layout.

- **Special permissions:** family baseline `dhiran.reports`; **Forfeiture** and **Profitability** are owner/manager-only; **KYC (Aadhaar/PAN/photo)** is a masked sensitive tier across the family with audited, gated reveal — stricter than any other report family.

None are GST-style filings, but Forfeiture + KYC are **rigid where regulated**.

**Net:** reuse the spine; register **Dhiran as a dedicated family** with the strictest sensitive-data posture in the product, owner/manager gating on Forfeiture & Profitability, KYC masking everywhere, and **two bespoke document templates** (Loan Statement, Forfeiture Notice/Record). Forfeiture is the one report that is special on all three axes — layout, permission, and compliance.
